## 3.1 Настройка серверного окружения и реализация backend-части

### 3.1.1 Выбор технологий серверной части

Для серверной части корпоративной системы управления выездным сервисом и ремонтом выбран стек **Node.js + Express + TypeScript**. В качестве СУБД на этапе разработки и отладки используется **SQLite** (файл `server/data/app.db`), что не требует отдельной установки сервера баз данных на рабочем месте разработчика. Для промышленной эксплуатации подготовлена эквивалентная схема **PostgreSQL** (`server/schema.postgresql.sql`), пригодная для включения в пояснительную записку (модель данных).

Дополнительно применяются:
- **jsonwebtoken** — аутентификация по токену JWT;
- **bcryptjs** — хеширование паролей;
- **socket.io** — push-обновления клиентов при смене статуса заявки;
- **better-sqlite3** — синхронный доступ к SQLite с высокой производительностью на малых нагрузках.

### 3.1.2 Структура каталогов backend

Серверная часть размещена в каталоге `server/`:

| Путь | Назначение |
|------|------------|
| `server/src/index.ts` | Точка входа, HTTP-сервер, подключение маршрутов и WebSocket |
| `server/src/db.ts` | Инициализация схемы БД |
| `server/src/migrate.ts` | Миграции (добавление полей: приоритет, телефон клиента, фото в истории) |
| `server/src/auth.ts` | Middleware проверки JWT и ролей |
| `server/src/roles.ts` | Матрица прав ролей `dispatcher` и `executor` |
| `server/src/routes/auth.ts` | Вход, профиль, смена пароля |
| `server/src/routes/requests.ts` | CRUD заявок, назначение, смена статуса, история |
| `server/src/routes/employees.ts` | Управление сотрудниками |
| `server/src/routes/stats.ts` | Сводная статистика для диспетчера |
| `server/src/seed.ts` | Заполнение демонстрационными данными |

### 3.1.3 Модель данных

В базе данных выделены три основные сущности:

1. **employees** — сотрудники (диспетчер / исполнитель): табельный номер, ФИО, хеш пароля, роль, признак активности.
2. **service_requests** — заявки на выезд: тема, клиент, адрес, телефон, описание, дата выезда, статус, приоритет, координаты, назначенный исполнитель.
3. **repair_history** — журнал изменений: переходы статусов, комментарий, опционально фото результата (Base64).

Жизненный цикл заявки: `new` → `assigned` → `in_progress` → `completed` (либо `cancelled`).

### 3.1.4 REST API

Сервер предоставляет REST API с префиксом `/api`. Основные конечные точки:

| Метод | URL | Назначение | Роль |
|-------|-----|------------|------|
| POST | `/api/auth/login` | Авторизация, выдача JWT | Все |
| GET | `/api/auth/me` | Данные текущего пользователя | Авторизованный |
| POST | `/api/auth/change-password` | Смена пароля | Авторизованный |
| GET | `/api/requests` | Список заявок (с фильтрами) | Диспетчер — все; исполнитель — только свои |
| POST | `/api/requests` | Создание заявки | Диспетчер |
| PATCH | `/api/requests/:id/assign` | Назначение исполнителя | Диспетчер |
| PATCH | `/api/requests/:id/status` | Обновление статуса и отчёта | Исполнитель / диспетчер |
| PATCH | `/api/requests/:id/cancel` | Отмена заявки | Диспетчер |
| GET | `/api/requests/:id/history` | История по заявке | Авторизованный |
| GET | `/api/stats` | Статистика | Диспетчер |
| GET | `/api/employees` | Список сотрудников | Диспетчер |

Параметры фильтрации списка заявок: `status`, `priority`, `search` (поиск по номеру, клиенту, адресу).

### 3.1.5 Авторизация и разграничение прав

После успешного входа клиент получает JWT и передаёт его в заголовке `Authorization: Bearer <token>`. Middleware `authRequired` проверяет подпись и срок действия токена. Дополнительно `requireRole('dispatcher')` ограничивает доступ к административным операциям.

Роль **executor** (исполнитель) может:
- просматривать только назначенные ему заявки;
- переводить заявку в статусы `in_progress` и `completed`;
- прикреплять текстовый отчёт и фотографию результата.

Роль **dispatcher** (диспетчер) может создавать заявки, назначать исполнителей, отменять заявки, просматривать статистику и управлять учётными записями.

### 3.1.6 Настройка и запуск серверного окружения

На рабочем месте установлены **Node.js** (LTS) и менеджер пакетов **npm**. Последовательность развёртывания:

```text
cd C:\Users\PC\vscode\DP\server
npm install
npm run seed
npm run dev
```

Команда `seed` создаёт файл БД и демо-учётные записи:
- диспетчер: `DISPATCHER` / `demo123`;
- исполнитель: `405IS` / `demo123`.

Сервер слушает порт **3001**. Проверка работоспособности: `GET http://localhost:3001/api/health` → `{"ok":true,"service":"field-service-api"}`.

### 3.1.7 Синхронизация в реальном времени

При изменении заявки сервер рассылает событие `sync` через **Socket.IO**. Это позволяет веб-панели диспетчера и мобильному клиенту обновлять списки без полной перезагрузки страницы. На мобильном клиенте дополнительно реализован резервный **polling** (опрос API каждые 20 с) на случай нестабильного Wi‑Fi.

## 3.2 Разработка мобильного приложения исполнителя

*(рекомендуемый заголовок: на React Native (Expo) и TypeScript)*

### 3.2.1 Обоснование выбора платформы разработки

Мобильный модуль исполнителя разработан с использованием **Expo** и **React Native**. Данный выбор обусловлен следующими факторами:
- единая кодовая база на языке **TypeScript** с веб-панелью диспетчера;
- возможность отладки на реальном устройстве через приложение **Expo Go** без установки Android Studio;
- наличие готовых модулей (камера, хранилище, навигация, deep links в карты);
- соответствие компетенциям по специальности 09.02.07 «Информационные системы и программирование».

Среда разработки — **Visual Studio Code**, запуск — `npm start` в корне проекта.

### 3.2.2 Архитектура мобильного клиента

Приложение построено по слоистой схеме:

| Слой | Каталог / файлы | Функция |
|------|-----------------|---------|
| UI (экраны) | `src/screens/` | LoginScreen, TaskListScreen, TaskDetailScreen, ProfileScreen |
| Состояние | `src/context/` | AuthContext, TasksContext |
| API | `src/api/` | HTTP-клиент, методы auth и requests |
| Утилиты | `src/utils/` | Подписи статусов, приоритетов, открытие карт |
| Навигация | `src/navigation/` | Stack-навигация React Navigation |
| Локальное хранилище | `src/storage.ts` | Сохранение JWT (AsyncStorage) |

Точка входа — `App.tsx`; конфигурация адреса API — `src/config.ts` и переменная окружения `EXPO_PUBLIC_API_URL` (файл `.env`).

### 3.2.3 Реализованные экраны

**Экран авторизации** (`LoginScreen.tsx`). Поля: табельный номер и пароль. После успешного входа выполняется запрос `POST /api/auth/login`; токен сохраняется в AsyncStorage для последующих сеансов.

**Список заданий** (`TaskListScreen.tsx`). Отображает заявки, назначенные текущему исполнителю. Реализованы:
- фильтры по статусу (все / назначены / в работе / выполнены);
- строка поиска;
- отображение приоритета (низкий, обычный, срочный);
- pull-to-refresh и автоматическое обновление по WebSocket.

**Карточка заявки** (`TaskDetailScreen.tsx`). Содержит:
- данные клиента и адрес;
- кнопку «Маршрут в картах» (внешнее приложение карт через `expo-linking`);
- звонок клиенту (`tel:`), если указан телефон;
- кнопки «Начать работу» и «Завершить с отчётом»;
- поле комментария;
- съёмку фото результата (`expo-image-picker`);
- блок «История по заявке» (запрос `GET /api/requests/:id/history`).

**Профиль** (`ProfileScreen.tsx`). Отображение ФИО, табельного номера, роли; форма смены пароля; выход из системы.

### 3.2.4 Клиентская логика работы с API

HTTP-запросы инкапсулированы в модуле `src/api/client.ts`. К каждому защищённому запросу добавляется заголовок `Authorization`. При смене статуса на «Выполнена» на сервер передаются поля `status`, `note` и при необходимости `photoBase64`.

Для работы в локальной сети в файле `.env` задаётся IP-адрес ПК разработчика, например:

```env
EXPO_PUBLIC_API_URL=http://192.168.0.104:3001
```

Телефон и компьютер должны находиться в одной сети (Wi‑Fi / Ethernet через общий роутер). В `app.json` для Android включён параметр `usesCleartextTraffic`, так как на этапе разработки используется протокол HTTP.

### 3.2.5 Вспомогательный модуль диспетчера

Для полноценной проверки мобильного модуля реализована веб-панель диспетчера (`dispatcher/`, React + Vite): создание заявок, назначение исполнителя, сводная статистика, управление сотрудниками. Панель не входит в мобильный модуль, но необходима для сквозных сценариев дипломного проекта.

## 3.3 Интеграция и отладка взаимодействия

### 3.3.1 Схема взаимодействия компонентов

Взаимодействие строится по клиент-серверной модели:

```text
[Смартфон: Expo Go] ←— Wi‑Fi LAN —→ [ПК: Node.js API :3001]
                                              ↑
[Браузер: диспетчер :5173] —— proxy ———————┘
```

Диспетчерская панель при разработке обращается к API через прокси Vite (`/api` → `localhost:3001`). Мобильный клиент обращается напрямую по IP ПК.

### 3.3.2 Типовой сценарий интеграции

1. Диспетчер создаёт заявку (`POST /api/requests`) с указанием приоритета и адреса.
2. Диспетчер назначает исполнителя (`PATCH /api/requests/:id/assign`) → статус `assigned`.
3. Сервер записывает событие в `repair_history` и отправляет `sync` по WebSocket.
4. Мобильное приложение получает обновление и отображает заявку в списке.
5. Исполнитель нажимает «Начать работу» → `PATCH .../status` → `in_progress`.
6. Исполнитель завершает работу с комментарием и фото → `completed`.
7. Диспетчер видит новый статус на панели без перезагрузки.

### 3.3.3 Проблемы при отладке и их устранение

| Проблема | Причина | Решение |
|----------|---------|---------|
| «Ошибка запроса» на панели диспетчера | Не запущен или устарел процесс `npm run server` | Перезапуск сервера, проверка `/api/health` |
| Мобильное приложение не подключается | В `.env` указан `localhost` вместо IP ПК | Заменить на IPv4 из `ipconfig` |
| Список заявок пуст | Исполнитель не назначен диспетчером | Назначить заявку на учётную запись `405IS` |
| 403 при входе | Вход диспетчера в мобильное приложение | Мобильный клиент только для роли `executor` |
| Нет обновления статуса | Обрыв WebSocket | Срабатывает polling; проверить одну сеть Wi‑Fi |

Отладка велась с помощью: логов терминала Node.js, вкладки Network в браузере, проверки `http://<IP>:3001/api/health` с телефона, инструментов Expo (Metro Bundler, QR-код).

### 3.3.4 Проверка согласованности данных

Каждое изменение статуса дублируется в таблице `repair_history`, что обеспечивает трассируемость и возможность анализа в пояснительной записке. Идентификатор исполнителя в JWT (`sub`) сопоставляется с полем `assignee_id` в заявке — попытка изменить чужую заявку блокируется кодом **403**.
